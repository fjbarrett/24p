# Project Context
- Architecture: Next.js 16 frontend with Google NextAuth; Rust API (Axum) handles profiles (usernames), lists (now keyed by username + slug), favorites, ratings, TMDB search/detail, and imports. Frontend calls the Rust API through `rustApiFetch` with a Next.js proxy at `/api/rust/*` for browser-safe same-origin requests; `/api/tmdb/*` rewrites to the same proxy for direct TMDB hits. Only auth routes remain in Next.js.
- Environment: `DATABASE_URL` (Neon Postgres, sslmode=require enforced), `DB_MAX_CONNECTIONS` defaults to 5, `STRAWBERRY_BASE_URL` required for TMDB/ratings, `RUST_API_ORIGIN`/`RUST_API_BASE_URL` target the Rust API for server calls (client defaults to the proxy), and `NEXT_PUBLIC_AUTH_CALLBACK_URL` controls the Google sign-in redirect. CORS is open on the Rust API. Google sign-in now returns users to `http://localhost:3000` after authentication. Docker defaults point the web app at `http://rust-api:8080` for Rust calls and `http://localhost:3000` for callbacks.
- Data/Features: Lists are private by default, scoped by `userEmail`, and use per-user slugs (`/username/list-name`). Owners can set list visibility to public once a username exists; public lists are readable by anyone. Profiles now include a public/private toggle, and public profiles render at `/username` with that user's public lists. Private lists can be shared with specific usernames for access without appearing in the public directory, and shared users can optionally be granted edit rights for collaboration. Users can favorite public lists and see them on their dashboard. List detail supports sorting by list/TMDB/IMDb/Letterboxd ratings and CSV export. Imports accept CSV with optional TMDB ids and user ratings. List cache uses localStorage/sessionStorage client-side.
- TMDB/External: All TMDB calls flow through the Strawberry proxy; IMDb/Letterboxd ratings and Apple TV links (via Strawberry CheapCharts) surface when an IMDb id exists.
- UI: Sticky centered search bar for signed-in users; search results now include artist hits linking to `/artists/:id` filmography pages alongside movie results with posters/title/year and a plus button to choose a list and add the movie inline. Signed-in home shows list actions, favorites, and profile/settings links; the profile page holds the username (vanity URL) setter, while settings currently hosts the list import flow. List editor now includes shared-access controls for private lists and redirects to the updated URL after renames. Signed-out splash shows a public list directory. Lists render as more compact dark cards with single-color accents and link to `/username/list-name`, and list detail posters are smaller in a denser mobile grid. Movie detail is mobile-first with full-width poster, community rating chips, borderless credits, and director/DP/top-billed cast links to IMDb.
- Branding: SEO metadata now presents the app as "24p" for titles, descriptions, and social previews.
- Cleanup: Removed unused demo components (feature grid, stats bar, share card, rating widgets) and obsolete sample data/diagrams to keep the footprint lean.
- Accessibility: Modals/dialogs labeled, search uses ARIA combobox/listbox with live updates, and form/rating interactions announce status messages.
- Tooling: Bun is the package manager; use `bun install`, `bun run dev`, `bun run build`, `bun run lint`, `bun run test` (once present). Next.js treats `jose` as a server external to avoid Turbopack/NextAuth `webcrypto.js` resolution errors. GitHub CLI (`gh`) is available for PR automation; a local Codex skill (`github-feature-flow`) lives under `~/.codex/skills` with a Keep a Changelog template reference.
- Deployment: Dockerfiles exist for the Next.js app (multi-stage Bun build, Node runtime) and Rust API, plus `docker-compose.yml` that boots Postgres, the API, and the web app; `.env.docker.example` seeds required env vars and defaults to ports 3000/8080/5432. Local compose uses `DB_SSLMODE=disable`; set to `require` when targeting a TLS-enabled Postgres.
